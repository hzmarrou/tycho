# Ontozense Independent Review — 2026-04-10

## Verdict
Proceed to Step 6 only after fixing two foundation issues: (1) remove/disable the legacy Source-A-to-13-fields path (`extract-dd` + `dd_extractor`) that violates the pivot, and (2) enforce router auto-dispatch safety (`--auto` confidence gating and true multi-source dispatch). The rest of the foundation is strong enough to support fusion once those are corrected.

## Strengths
- The architectural intent is clear and unusually well-documented. `docs/PLAYBOOK.md` is concrete, operational, and testable.
- Source A SPIRES bypass is implemented in the right place and shape: `raw_completion_output` is parsed first, with fallback to `extracted_object` only when needed (`src/ontozense/extractors/domain_doc_extractor.py:195`, `src/ontozense/extractors/domain_doc_extractor.py:211`).
- Confidence honesty for key edge cases is materially improved: name-only concepts get a missing-definition penalty (~0.475) (`src/ontozense/extractors/domain_doc_extractor.py:354`) and both-missing-endpoint relationships score 0.30 (`src/ontozense/extractors/domain_doc_extractor.py:418`).
- Domain-neutrality protection is explicit and enforced (`tests/test_domain_neutrality.py:61`). I also re-ran it successfully.
- Source D deterministic output is shaped for later AI-RBX steps (referenced symbols + provenance): `src/ontozense/extractors/code_extractor.py:82`, `src/ontozense/extractors/code_extractor.py:274`, `src/ontozense/extractors/code_extractor.py:300`.
- Source B scaffold API is lockable/consumable for Step 6 and correctly advertises incompleteness with `NotImplementedError` (`src/ontozense/extractors/governance_extractor.py:155`).

## Issues
1. **[blocker] Methodological pivot leak is still first-class in shipped CLI/API**  
   What: The old “extract rich DD directly from Source A” path is still exposed as a top-level command (`extract-dd`) and full extractor (`dd_extractor`). That directly contradicts the pivot (“fusion as output, not extraction target”).  
   Where: `src/ontozense/cli.py:30`, `src/ontozense/extractors/dd_extractor.py:1`, `src/ontozense/extractors/dd_extractor.py:127`, `src/ontozense/extractors/__init__.py:3`.  
   Recommendation: Hard-disable or remove `extract-dd` now (or rename to `legacy-extract-dd` with explicit warning and non-default install path). Keep only Source A concept/relationship extraction as supported route into Step 6.

2. **[major] `ingest --auto` does not enforce the documented confidence gate (>0.9)**  
   What: Auto mode dispatches all routed A files, regardless of confidence. PLAYBOOK §5 says auto-route only when confidence > 0.9.  
   Where: `src/ontozense/cli.py:371`, `src/ontozense/cli.py:381`.  
   Recommendation: Filter `decisions` to `confidence > 0.9` before dispatch; emit explicit skipped-low-confidence list.

3. **[major] Multi-source routing is detected but not actually dispatched as multi-source**  
   What: Router can return A+D for markdown, but ingest dispatch logic routes by `primary_source` only. A+D ends up dispatched only to A.  
   Where: `src/ontozense/router/router.py:24`, `src/ontozense/router/router.py:547`, `src/ontozense/cli.py:381`, `src/ontozense/cli.py:402`.  
   Recommendation: Dispatch per `decision.sources`, not only `primary_source`; log each dispatch leg with same source file and run id.

4. **[major] Source A confidence rubric is not an exact implementation of PLAYBOOK §3**  
   What: Implementation is mostly narrative-overlap scoring with custom additions; it does not implement full field-category rubric exactly as documented (ENUM/CITATION/CATEGORICAL/STRUCTURED categories and values). Regex enrichment also injects non-rubric values (e.g., 0.85, 0.4).  
   Where: `src/ontozense/extractors/domain_doc_extractor.py:451`, `src/ontozense/cli.py:725`, `src/ontozense/cli.py:759`.  
   Recommendation: Either (a) align code to rubric table exactly, or (b) update PLAYBOOK §3 to reflect real scoring policy; do not leave them divergent.

5. **[major] Exit-code-3 logic for Source A checks concepts only, not all extracted elements**  
   What: Low-confidence failure gating uses concept count only; relationships are ignored. This can incorrectly fail/pass relative to stated “all elements low confidence” behavior.  
   Where: `src/ontozense/cli.py:614`, `src/ontozense/cli.py:673`.  
   Recommendation: Compute quality gate over both concepts and relationships (or explicitly codify “concept-only gate” in PLAYBOOK and CLI help).

6. **[major] Default model inconsistency (library default still `gpt-5.2`)**  
   What: CLI default is `azure/gpt-5.4`, but `DomainDocumentExtractor` constructor default remains `azure/gpt-5.2`. Non-CLI callers silently downgrade output quality.  
   Where: `src/ontozense/cli.py:445`, `src/ontozense/extractors/domain_doc_extractor.py:133`.  
   Recommendation: Make extractor default match CLI/playbook (`gpt-5.4`) or require explicit model everywhere.

7. **[major] High-value coverage gap: no automated tests for Source D extractor behavior**  
   What: `code_extractor.py` has no dedicated `tests/test_code_extractor.py`, despite being Step 5-complete and structurally important for Step 6.  
   Where: `src/ontozense/extractors/code_extractor.py:1`; test suite has no corresponding file.  
   Recommendation: Add tests for Python/SQL extraction boundaries, citation regex hits, symbol extraction, parse-failure behavior, and provenance completeness.

8. **[minor] High-value coverage gap: CLI failure modes / routing safety behavior are untested**  
   What: No tests asserting exit code 2/3 semantics for `extract-a`, nor `ingest --auto` confidence gating behavior.  
   Where: `src/ontozense/cli.py:596`, `src/ontozense/cli.py:674`, `src/ontozense/cli.py:371`.  
   Recommendation: Add Typer CLI integration tests with mocked extractors/router decisions.

9. **[minor] Relationship provenance snippet anchors only on subject**  
   What: If subject is absent but object is present in source, snippet may be empty although evidence exists.  
   Where: `src/ontozense/extractors/domain_doc_extractor.py:432`.  
   Recommendation: Fallback to object anchor if subject snippet not found (mirror concept fallback style).

10. **[nit] Playbook internal drift note**

    Where: `docs/PLAYBOOK.md:309`, `docs/PLAYBOOK.md:333`.

## Answers to the 10 questions in §6
1. **Is the four-source architecture sound?**  
   Yes. It is the right decomposition for provenance-honest ontology bootstrapping. The missing dimension is not a fifth source but **cross-source identity resolution policy** (naming/alias normalization) as a first-class fusion concern.

2. **Is the fusion-as-output pivot correctly implemented?**  
   Partially. Source A/B/D implementations mostly respect it, but the legacy `extract-dd` path violates it and is still publicly exposed.

3. **Is the playbook the right level of abstraction?**  
   Mostly yes. It is close to executable policy. But the confidence rubric and auto-routing safety policy must be kept in lockstep with code; currently they drift.

4. **Is the field-aware confidence rubric defensible?**  
   Conceptually yes. Numerically, yes as a starting point. Practically, current code does not fully implement the documented categories, so defensibility is reduced by mismatch rather than by the numbers themselves.

5. **Does the SPIRES `raw_completion_output` bypass scare you?**  
   Moderately. It is a valid tactical workaround, but brittle to output-shape drift. Add contract tests with archived real outputs and a parser-version sentinel (fail loudly when parse yield suddenly collapses).

6. **Is AI-RBX deterministic-first preserved correctly?**  
   Yes for step 1. The deterministic output contains referenced symbols and provenance, so steps 2/3 are addable without reparsing. I do not see domain-locking assumptions in this layer.

7. **Is the router going to scale?**  
   Extension + sniffing scales operationally for MVP. The first breakage point will be ambiguous mixed artifacts and false positives as corpus heterogeneity grows. You’ll need decision telemetry and periodic heuristic tuning before introducing L3.

8. **Is canonical CSV for Source B sensible?**  
   Yes for MVP and governance trustworthiness. Do not add fuzzy header matching now. Provide a converter utility outside core parser if onboarding friction becomes a blocker.

9. **Is per-domain `log.md` useful or theatre?**  
   Useful for current single-user workflow. It is pragmatic and auditable. For SaaS/multi-process, add JSONL alongside it (or migrate) for machine analytics and concurrency-safe ingestion.

10. **Are you ready for Step 6?**  
   Almost. Fix the two blockers first: deprecate/remove legacy `extract-dd` path and enforce routing auto-dispatch safeguards. Then proceed.

## Things I would do differently
- I would not keep the legacy direct-DD extraction command enabled in the same CLI namespace once the pivot is adopted.
- I would make policy-code drift impossible by encoding key playbook constants (routing auto threshold, confidence rubric table) in one shared config module referenced by both docs generation and runtime.
- I would add a minimal “fusion-readiness contract test” suite now, before writing fusion logic (shape/provenance/confidence invariants per source).
