# Source D v1.2 Executable Rule Extraction — Design

**Status:** Draft, revision 2 (coder review questions on serialization and C/D overlap addressed). Brainstormed 2026-05-19.
**Target release:** v1.2.
**Scope owner:** `Source D` only; `Source A/B/C` semantics unchanged except where fusion consumes new `Source D` fields.
**Predecessor:** [docs/superpowers/specs/2026-05-17-source-cd-seeders-design.md](/mnt/c/Users/hzmarrou/OneDrive/python/projects/ontozense/docs/superpowers/specs/2026-05-17-source-cd-seeders-design.md) (v1.1 shipped `Source D` as a deterministic candidate seeder).

---

## 1. Goal

Upgrade `Source D` from a shallow code-shape seeder into a **domain-agnostic extractor of executable ontology evidence**.

In one line: **make `Source D` contribute evidence-backed executable rules for entities and attributes, not just code-shaped hints such as classes, methods, and validator function names.**

`Source D` remains an ontology seeder inside `survey`. It does **not** become a BRD generator, CRUD reporter, lineage product, or analyst-facing reverse-engineering system.

---

## 2. Why

v1.1 proved that code can seed useful candidate-graph concepts, but the extraction boundary is still too shallow for code-led and pipeline-led domains:

- classes seed entities
- fields seed attributes
- enums seed vocabularies
- methods seed behaviors
- top-level `validate_*` / `check_*` / `assert_*` functions seed weak rules

That is helpful, but it does not recover the actual executable constraints that domain code often contains:

- field validations
- cross-field constraints
- defaulting rules
- derivation formulas
- status transitions
- guarded eligibility logic
- row-rejection and quarantine rules in pipelines

The intended three-way framing is:

> Source A and B seed stated semantic truth.
> Source C seeds structural truth.
> Source D seeds executable rule truth.

For many domains, especially data-heavy ones, the code is where a meaningful share of the domain's actual business constraints live. If `Source D` cannot extract them, the ontology remains materially incomplete.

---

## 3. Product Boundary

### 3.1 In scope

`Source D v1.2` stays inside `survey` and emits ontology-seeding candidates only.

In scope:

- Python-first extraction
- shape-adaptive extraction across multiple code-style families
- deterministic extraction of entities, attributes, vocabularies, behaviors, and executable rules
- a structured `rule_payload` nested inside the existing candidate envelope
- shared anchoring and suppression logic
- optional LLM normalization after deterministic extraction only
- fusion-time consumption of the new rule structure

### 3.2 Out of scope

Out of scope:

- BRD generation
- CRUD matrices
- standalone lineage reports
- analyst-facing reverse-engineering outputs
- chat/explanation features over code
- `draft.owl` rule serialization changes
- new SHACL or SWRL emitters
- LLM-invented rules
- whole-program semantic summarization
- non-Python language support in v1.2

This is where the AI-RBX article is informative but not load-bearing. Its extraction discipline is relevant; its artifact scope is not.

## 4. Design Principles

### 4.1 Domain-agnostic, not NPL-specific

`Source D` must not hardcode NPL, banking, insurance, healthcare, or any other domain semantics. It should extract **generic executable rule primitives** and leave domain interpretation to fusion with Sources A/B/C.

### 4.2 Shape-adaptive, not class-first or pipeline-first

Different domains express business logic differently. Some repos are OO-heavy; others are function-heavy or pipeline-heavy. `Source D` must support multiple code-shape families in the same run.

### 4.3 Deterministic first, optional AI second

Deterministic extraction is required. Optional LLM use is allowed only to:

- improve phrasing
- normalize wording
- align synonyms

The LLM must never be the sole source of a promoted rule candidate.

### 4.4 Suppress ambiguity rather than over-promote noise

Ambiguous, technical, or weakly anchored findings should be emitted to audit as suppressed candidates rather than promoted into the candidate graph.

## 5. High-Level Architecture

`Source D v1.2` is a six-stage shape-adaptive pipeline:

```text
Python source ──▶ parse ──▶ dispatch ──▶ lift into shared IR ──▶ anchor/filter ──▶ emit candidates
                                                             │
                                                             ▼
                                                  optional LLM normalization
                                                  (phrasing only, evidence-backed)
```

Stage definitions:

- **Parse** — Python AST plus lightweight symbol/context extraction.
- **Dispatch** — inspect code shapes and run one or more extractor families.
- **Lift** — convert parser output into a shared intermediate representation (IR).
- **Anchor/filter** — resolve entity/attribute attachment; suppress technical or weakly anchored items.
- **Emit** — produce standard candidate records with optional `rule_payload`.
- **Normalize** — optional LLM rewriting of labels/phrasing only.

## 6. Extractor Families

A single repository may contain all of the following. `Source D` should run every applicable extractor and merge their findings through the shared IR.

### 6.1 `model_extractor`

Targets:

- classes
- dataclasses
- ORM models
- Pydantic models
- typed objects
- enums
- validator methods
- guarded methods with explicit constraints

Primary outputs:

- entities from models/classes
- attributes from typed fields/properties
- vocabularies from enums
- behaviors from non-trivial domain methods
- rules from explicit guards, validations, derivations, defaults, and transitions inside model-local code

### 6.2 `pipeline_extractor`

Targets:

- schema declarations
- DataFrame operations
- SQL embedded in Python
- task/DAG step functions when they carry domain transforms
- assertions, reject/quarantine logic
- derived columns
- state/status mappings in transforms

Primary outputs:

- entities from explicit datasets, schema objects, domain-bearing tables/views/topics, or stable DataFrame-like domain shapes
- attributes from declared or referenced columns/fields
- rules from filters, guards, derived columns, state mappings, reject/quarantine conditions, and explicit transformation predicates
- dependencies from joins/lookups/config/constant usage when directly evidenced

### 6.3 `procedural_extractor`

Targets:

- plain functions/modules
- guard clauses
- assignments/formulas
- dict/payload field access
- explicit state mutations
- helper-level validations/defaults

Primary outputs:

- entities when clearly evidenced by payload/schema/function context
- attributes from field/key access
- rules from explicit comparisons, assignments, defaults, and transitions
- behaviors from domain-significant functions

## 7. Shared Intermediate Representation

Before candidate emission, all extractor families should lift their findings into a shared IR.

### 7.1 IR node families

- `EntityFact`
- `AttributeFact`
- `VocabularyFact`
- `BehaviorFact`
- `RuleFact`

### 7.2 `RuleFact` contract

Minimum fields:

- `rule_kind`
- `subject_entity`
- `subject_attribute`
- `predicate`
- `object_value`
- `condition`
- `depends_on`
- `expression`
- `evidence_span`
- `code_context`
- `confidence`
- `extractor_family`

Notes:

- `subject_entity` and `subject_attribute` may be unresolved in raw IR, but must be resolved or intentionally suppressed before final emission.
- `evidence_span` should be compact and traceable, not a whole file dump.
- `confidence` is deterministic and rule-based in v1.2; it is not an LLM confidence score.

## 8. Candidate Schema

### 8.1 Outer candidate envelope

`Source D` continues to emit the common candidate envelope used by `candidate-graph.json`.

This section defines the **shared rule contract** for executable constraints. Although this document is `Source D`-focused, the `rule_payload` contract is intentionally cross-source: any source that emits explicit rule-shaped evidence in v1.2 must use the same payload shape if that evidence is meant to participate in rule fusion.

For non-rule candidates:

- `artifact_kind != "rule"`
- `rule_payload = null`

For rule candidates:

- `artifact_kind = "rule"`
- `rule_payload` populated

### 8.2 `rule_payload`

Recommended v1.2 shape:

```json
{
  "rule_kind": "validation",
  "subject_entity": "Payment",
  "subject_attribute": "amount",
  "predicate": "gt",
  "object_value": 0,
  "condition": null,
  "depends_on": [],
  "expression": "amount > 0",
  "evidence_span": "if amount <= 0: raise ValueError(...)",
  "normalization_status": "deterministic"
}
```

Field meanings:

- `rule_kind` — closed enum; see §8.3.
- `subject_entity` — primary entity the rule governs.
- `subject_attribute` — optional attribute the rule constrains or derives.
- `predicate` — normalized relation/operator name such as `gt`, `lte_other`, `required`, `in_set`, `derived_from`, `transitions_to`.
- `object_value` — literal, set, target state, referenced attribute, or structured expression target as applicable.
- `condition` — optional guard under which the rule applies.
- `depends_on` — referenced attributes, constants, config values, or other direct dependencies.
- `expression` — normalized machine-readable summary of the extracted logic.
- `evidence_span` — compact code excerpt backing the rule.
- `normalization_status` — `deterministic` or `llm_rephrased`.

### 8.3 `rule_kind` enum

Closed enum for v1.2:

- `validation`
- `derivation`
- `defaulting`
- `eligibility`
- `transition`
- `calculation`
- `dependency`

This taxonomy is deliberately domain-agnostic. It names the shape of the executable rule, not its business domain.

### 8.4 Source C compatibility

Explicit rule-shaped constraints from `Source C` should emit the same `rule_payload` contract wherever the structural evidence can be mapped cleanly.

Examples:

- `CHECK amount > 0` → `validation`
- `NOT NULL` → `validation` with `predicate=required`
- other explicit structural constraints may participate when they can be represented without source-specific hacks

This is a fusion contract decision, not a scope expansion into arbitrary new `Source C` inference. The rule is:

- if `Source C` already extracts an explicit constraint and that constraint is rule-shaped, it should use compatible `rule_payload`
- if `Source C` has no explicit constraint extraction for a construct, v1.2 does not require inventing it

This avoids ad hoc duplicate suppression in `Source D` and allows structurally equivalent constraints from `Source C` and `Source D` to merge through shared payload identity.

## 9. Deterministic Extraction Scope

v1.2 supports only high-confidence, local, deterministic patterns.

### 9.1 Supported rule patterns

`validation`

- null/missing checks
- direct comparisons: `>`, `>=`, `<`, `<=`, `==`, `!=`
- range checks: `min <= x <= max`
- membership checks: `x in {...}`, `x not in {...}`
- explicit rejection via `raise`, `assert`, or deterministic failure return

`derivation` / `calculation`

- direct assignments from expressions
- arithmetic formulas
- computed booleans
- simple deterministic normalization transforms

`defaulting`

- assign a default when a value is absent/null/empty
- constructor/factory defaults when explicit
- obvious fallback patterns

`eligibility`

- action/outcome proceeds only when an explicit predicate holds
- guarded allow/deny logic with clear semantic target

`transition`

- explicit state/status mutation under a guard
- conditionally assigned lifecycle values

`dependency`

- direct dependency on another attribute, constant, enum member, config value, or lookup result when the code makes that relationship explicit

### 9.2 Not supported in v1.2

Not supported:

- long interprocedural reasoning across call chains
- portfolio-wide lineage
- inferred business purpose of a module or workflow
- hidden semantics from large imperative blocks with ambiguous targets
- rules inferred only from names
- runtime-only behavior requiring execution or dynamic tracing

## 10. Anchoring Rules to Ontology Subjects

A rule may only be promoted when it is anchored to ontology-relevant subjects.

### 10.1 Anchoring order

Anchor in this order:

1. enclosing class/entity context
2. explicit field/attribute context
3. local symbol context with dependency-only fallback
4. otherwise suppress to audit

### 10.2 Promotion threshold

Every non-suppressed rule candidate must be anchored to at least one of:

- a `subject_entity`, or
- a clearly domain-bearing `subject_attribute`

If neither can be established deterministically, the rule is not ontology-grade yet and must be suppressed.

### 10.3 Suppression reasons

Common suppression categories:

- infrastructure/framework plumbing
- test/mocks/generated code
- generic technical assertions with no domain-bearing subject
- ambiguous anchoring across multiple equally plausible subjects
- weak extraction patterns below the configured confidence threshold

## 11. Fusion Semantics

`Source D` is executable evidence, not naming authority.

Fusion principles:

- `Source A/B` remain primary for business wording and semantic labels
- `Source C` remains primary for structural confirmation
- `Source D` contributes executable constraints, derivations, defaults, and transitions
- explicit rule-shaped constraints from `Source C` participate in the same rule-fusion contract when they emit compatible `rule_payload`
- `Source D`-only rules are allowed when evidence is strong enough; they remain clearly marked as executable-only evidence until corroborated
- conflicts between `Source D` and `Source A/B` must not be silently collapsed

### 11.1 Rule merge identity

Rule merge identity should rely on structured `rule_payload`, not only human-readable `label` text.

At minimum, matching should consider:

- `rule_kind`
- `subject_entity`
- `subject_attribute`
- `predicate`
- normalized `object_value`
- normalized `condition` when present

Text labels are display material and may be LLM-rephrased. They must not be the only merge key.

When `Source C` and `Source D` emit structurally equivalent rules, duplicate handling must happen through this shared merge identity. It must not rely on ad hoc suppression logic inside `Source D`.

### 11.2 Serialization boundary for v1.2

v1.2 includes executable-rule extraction and fusion, but **does not** extend downstream rule serialization.

For v1.2:

- `rule_payload` is first-class in `candidate-graph.json`
- fusion consumes and preserves structured rule evidence
- `draft.owl` emission remains unchanged for executable rules
- no SHACL emitter is added
- no SWRL emitter is added

This means v1.2 establishes the extraction and fusion contract first. OWL/SHACL/SWRL serialization is explicitly deferred to a later design once the real fused rule inventory is available.

---

## 12. LLM Role

LLM use is optional and second-pass only.

Allowed uses:

- rewrite labels into cleaner ontology phrasing
- normalize wording across extractors
- align obvious synonyms after deterministic extraction
- improve readability of promoted rule labels

Disallowed uses:

- invent rules
- invent entities or attributes
- override deterministic evidence
- promote a rule candidate that lacks deterministic evidence
- change merge identity independently of the evidence-backed payload

This keeps `Source D` useful in offline/CI settings and prevents the LLM from becoming the hidden source of truth.

## 13. AI-RBX Decision

Decision:

- keep AI-RBX as a **design input**
- drop AI-RBX as a **product template**

### 13.1 What to borrow from AI-RBX

Borrow:

- deterministic parsing first
- enriched intermediate representation
- strict traceability back to code
- structured outputs
- optional post-extraction AI normalization

### 13.2 What not to import

Do not import:

- BRD generation
- CRUD outputs
- standalone lineage reports
- analyst-facing reverse-engineering workflow
- broad system explanation as a `Source D` responsibility

AI-RBX is therefore a methodological reference, not the v1.2 scope document.

## 14. Acceptance Criteria

The implementation is complete when:

- **AC1 — common contract:** All `Source D` extractor families emit the existing candidate envelope, and rule candidates populate `rule_payload` while non-rule candidates leave it null.
- **AC1a — cross-source rule contract:** Explicit rule-shaped constraints from `Source C` that are already extracted and structurally representable also emit compatible `rule_payload`, so they can merge with equivalent `Source D` rules.
- **AC2 — shape-adaptive extraction:** A Python repo containing model-centric, pipeline-centric, and procedural code yields candidates from every applicable extractor family in one `survey` run.
- **AC3 — deterministic rule extraction:** Explicit validations, derivations, defaults, eligibility checks, and transitions are extracted without requiring any LLM.
- **AC4 — anchored promotion:** Every non-suppressed rule candidate is anchored to at least a subject entity or a clearly domain-bearing subject attribute. Unanchored rules go to audit with explicit suppression reasons.
- **AC5 — pipeline support:** Pipeline-style code can seed entities, attributes, and rules from schemas, column references, filters, derived columns, and reject/assert logic without requiring classes.
- **AC6 — model support:** Model-centric code continues to seed entities, fields, enums, behaviors, and explicit validations correctly under the new architecture.
- **AC7 — procedural support:** Procedural code can seed rules when explicit deterministic patterns are present, without requiring classes or DataFrame abstractions.
- **AC8 — provenance:** Every promoted or suppressed `Source D` candidate includes source artifact and evidence sufficient to trace back to the exact code location responsible for the extraction.
- **AC9 — optional AI only:** Disabling the optional LLM normalization pass leaves deterministic extraction fully functional and does not reduce the set of evidence-backed promoted rules.
- **AC10 — fusion-ready:** `Source D` outputs can be fused with `Source A/B/C` using the agreed `rule_payload` and anchoring semantics without ad hoc downstream hacks.
- **AC11 — serialization boundary preserved:** `draft.owl` output remains unchanged for executable rules in v1.2, and no SHACL/SWRL emitter is introduced as part of this design.

## 15. Open Questions for Implementation Planning

These are implementation-planning questions, not design blockers:

1. Which Python pipeline frameworks merit first-class deterministic support in v1.2: pandas only, pandas + PySpark, or a broader adapter list?
2. Should SQL embedded inside Python be handled by a shared `sqlglot` helper reused from `Source C`, or by a narrower `Source D`-local extractor first?
3. Should `depends_on` store only canonical ontology-aligned names, or both raw local symbols and canonicalized names?
4. Where should conflict markers for contradictory executable vs stated semantics live: candidate audit entries, provenance, or a dedicated conflict block?

## 16. Scope Summary

In scope for v1.2:

- Python-first shape-adaptive `Source D`
- shared IR with `RuleFact`
- nested `rule_payload` for rule candidates
- compatible `rule_payload` for explicit rule-shaped `Source C` constraints that already exist
- deterministic extraction across model/pipeline/procedural code
- shared anchoring and suppression logic
- optional LLM phrasing normalization only
- fusion consumption of executable rule evidence

Out of scope for v1.2:

- non-Python language support
- portfolio-wide lineage products
- BRDs / CRUD / analyst reports
- interprocedural whole-program inference
- runtime tracing
- LLM-generated rule discovery
- OWL/SHACL/SWRL rule serialization
