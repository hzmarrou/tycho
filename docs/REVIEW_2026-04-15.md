# Ontozense Tester-Readiness Review — 2026-04-15

## Verdict
Not ready for external tester handoff yet. A competent Python user can install and run parts of the flow, but the current Windows CLI experience has hard crashes in normal commands due non-ASCII console characters, and the tutorial expected outputs no longer match real behavior in key steps. Those two issues together create a high chance the tester stops early or loses trust in the walkthrough.

## What works well
- Fresh install worked in a new Linux venv: `pip install -e ".[dev]"` completed successfully.
- Test baseline is strong at the module level: `pytest -q` passed (`354 passed`).
- Core fusion/lint/query/file-back pipeline mechanics are implemented and functionally connected.
- Domain-neutrality guard remains effective (`tests/test_domain_neutrality.py` passed, and no banned terms matched in `src/ontozense/`).

## Blockers (must fix before handoff)
1. **[blocker] Windows CLI crashes on common commands because output uses Unicode symbols not encodable by cp1252.**
   - `ingest` prints `→`/`↓` markers, which crash on Windows code pages at runtime (`src/ontozense/cli.py:133`).
   - `extract-a` prints `≥` in summary labels and can crash after writing outputs (`src/ontozense/cli.py:514`, `src/ontozense/cli.py:515`).
   - Also uses `✗`/`⚠` in other paths (`src/ontozense/cli.py:439`, `src/ontozense/cli.py:470`), so this is systemic, not a single line.

2. **[blocker] Tutorial expected routing output is wrong vs actual router behavior.**
   - Tutorial says governance JSON routes to Source B and expects `A=1, B=1, D=5` (`docs/ontozense-npl-tutorial.md:141`, `docs/ontozense-npl-tutorial.md:143`, `docs/ontozense-npl-tutorial.md:147`).
   - Router maps `.json` to Source C by default (`src/ontozense/router/router.py:120`) and has no JSON content-sniff override path (`src/ontozense/router/router.py:342`, `src/ontozense/router/router.py:350`).
   - In practice this yielded `A=1, C=2, D=4` in the tutorial workspace run.

3. **[major] Tutorial query example uses `"Default"`, but shipped governance data does not include `Default`.**
   - Tutorial positions `Default` as the canonical query walk-through (`docs/ontozense-npl-tutorial.md:337`, `docs/ontozense-npl-tutorial.md:343`).
   - Shipped governance fixture starts with `Borrower`, `Collateral`, etc., and no `Default` entry (`docs/governance_example.json:4`, `docs/governance_example.json:11`, `docs/governance_example.json:67`).
   - Result: tutorial-style `query "Default"` returned no matches in the fused output.

4. **[major] Lint structural-gap output is far too noisy for first-time tester workflow.**
   - Structural-gap logic emits one warning per weak community pair (`src/ontozense/core/lint.py:250`, `src/ontozense/core/lint.py:259`, `src/ontozense/core/lint.py:305`).
   - CLI prints every finding verbatim (`src/ontozense/cli.py:892`, `src/ontozense/cli.py:910`, `src/ontozense/cli.py:913`).
   - On shipped tutorial fused data this produced `377` structural-gap warnings and `385` total warnings, overwhelming actionable signal.

5. **[major] `suggest-bridges` scales poorly because it performs one LLM call per structural gap.**
   - The command derives all holes (`src/ontozense/cli.py:589`, `src/ontozense/cli.py:590`) then calls bridge generation over every pair (`src/ontozense/cli.py:609`, `src/ontozense/cli.py:612`).
   - Bridge generation loops hole-by-hole and calls `_call_llm` each iteration (`src/ontozense/core/bridging.py:96`, `src/ontozense/core/bridging.py:101`).
   - With hundreds of holes, this becomes impractical for a tester run.

6. **[major] `extract-a` failure path can surface raw traceback instead of a clean user-facing CLI error.**
   - `extractor.extract_from_file()` is called without command-level exception handling (`src/ontozense/cli.py:350`).
   - Under model/auth/deployment failure, exceptions from OntoGPT propagate (`src/ontozense/extractors/ontogpt_extractor.py:287`, `src/ontozense/extractors/ontogpt_extractor.py:288`) and appear as traceback output, which is not a controlled UX.

7. **[minor] README is stale for current tester flow.**
   - Quick Start still describes legacy `extract/refine/export` flow only (`README.md:15`, `README.md:16`, `README.md:17`), while tester assignment centers on `extract-a/fuse/lint/suggest-bridges/query/file-back`.

## Tester experience notes
- The tester can install and run tests, and core dataflow components do work.
- The first likely failure on Windows is very early: `ingest --dry-run` can crash before finishing output due arrow glyphs.
- If they move past that, tutorial expected outputs diverge from actual routing and query results, forcing guesswork.
- Lint then produces hundreds of structural-gap warnings; signal-to-noise is poor.
- If they try `suggest-bridges` immediately after this, runtime/cost balloons because prompt calls scale with gap count.
- Credential onboarding is fragile: environment loading is silent (`src/ontozense/cli.py:19`) and the tutorial carries the burden of manual variable setup (`docs/ontozense-npl-tutorial.md:31`).
- Python 3.10 support is declared but was not runtime-verified in this environment (`pyproject.toml:10`, `pyproject.toml:51`).

## Answers to the 8 questions in §6
1. **Will a competent Python user who has never seen this repo be able to run the tutorial end-to-end without asking a human?**
No. Biggest blocker is Windows CLI Unicode crashes in baseline tutorial commands (`src/ontozense/cli.py:133`, `src/ontozense/cli.py:514`). After that, tutorial output mismatches create additional confusion (`docs/ontozense-npl-tutorial.md:141`, `src/ontozense/router/router.py:120`).

2. **Are there cliff-edges a new user would hit?**
Yes.
- UnicodeEncodeError cliff-edge on Windows terminals (`src/ontozense/cli.py:133`, `src/ontozense/cli.py:515`).
- Tutorial assumptions that no longer match code (`docs/ontozense-npl-tutorial.md:143`, `docs/ontozense-npl-tutorial.md:337`).
- Raw traceback exposure on extract failures (`src/ontozense/cli.py:350`, `src/ontozense/extractors/ontogpt_extractor.py:288`).

3. **Is the 354-test suite covering what matters, or are there critical paths with no tests?**
Core modules are well covered; CLI coverage is uneven.
- CLI tests exist for `ingest` and `extract-a` (`tests/test_cli_ingest.py:24`, `tests/test_cli_extract_a.py:30`).
- Critical handoff commands (`fuse`, `lint`, `suggest-bridges`, `query`, `file-back`) are exercised mainly via underlying module tests, not equivalent CLI integration depth. This gap is visible because major CLI regressions (Windows encoding crashes) still passed suite-wide.

4. **Does `docs/ontozense-npl-tutorial.md` accurately describe current code behavior?**
Not fully.
- Routing expectations are stale (`docs/ontozense-npl-tutorial.md:141` vs `src/ontozense/router/router.py:120`).
- Query walkthrough for `Default` mismatches shipped fixture content (`docs/ontozense-npl-tutorial.md:337` vs `docs/governance_example.json:4`).
- Lint expected warning scale (`docs/ontozense-npl-tutorial.md:317`) is far below observed output on shipped fused example.

5. **Is confidence scoring still honest and aligned with PLAYBOOK §3?**
Mostly improved and largely aligned, with one caveat.
- Playbook says element confidence is an average of field-level evidence (`docs/PLAYBOOK.md:103`, `docs/PLAYBOOK.md:105`).
- Fusion computes element confidence as an average over field provenance (`src/ontozense/core/fusion.py:111`, `src/ontozense/core/fusion.py:116`).
- Caveat: Source A merge currently reuses concept-level confidence for multiple fields (`src/ontozense/core/fusion.py:242`, `src/ontozense/core/fusion.py:247`, `src/ontozense/core/fusion.py:250`), which reduces per-field granularity.

6. **Does `suggest-bridges` justify its existence?**
Potentially yes, but not in current default flow shape.
- Concept is useful when structural gaps are curated and limited.
- Current implementation on raw lint output is operationally heavy (one LLM call per hole: `src/ontozense/core/bridging.py:96`, `src/ontozense/core/bridging.py:101`), so it risks feeling like theatre/noise for a first tester run.

7. **Is there anything you’d strongly recommend cutting before handoff?**
Yes: cut `suggest-bridges` from the default tutorial path until hole output is bounded/prioritized.
- Keep command available, but move it to an “advanced/optional” section after filtering structural gaps.

8. **What’s the one thing most likely to make the tester give up in frustration?**
The first Unicode crash in `ingest`/`extract-a` on Windows terminals (`src/ontozense/cli.py:133`, `src/ontozense/cli.py:515`). It happens during basic success-path usage and looks like tooling instability.

## What I'd cut
- Cut `suggest-bridges` from the primary tutorial path until structural-gap volume is controlled.
- Cut tutorial “Expected output” blocks that currently over-specify stale counts/labels; replace with invariant checks (for example: “fused output exists”, “governance_validated > 0”, “query returns at least one known term”).

## What I'd add
- Add an explicit Windows-safe console output policy for CLI (ASCII-only symbols in default output).
- Add CLI integration tests for `fuse`, `lint`, `query`, `file-back`, and `suggest-bridges` in Windows encoding conditions.
- Add a pre-handoff tutorial smoke test in CI that executes documented commands against shipped fixtures and verifies the expected outcomes stay synchronized with docs.
